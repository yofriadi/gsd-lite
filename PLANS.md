# gsd-lite (renamed to gpd, get-pi-done) → gsd-like plan+build framework

## Objective

Redesign the `gsd-lite` repo to a full **plan + build** framework on the Pi SDK:

- add a lightweight `/init` bootstrap that is the sole writer of `PROJECT.md`,
- rework `/plan` to produce a phase/requirements artifact hierarchy and to maintain the living `REQUIREMENTS`/`ROADMAP`/`STATE` docs,
- add a sequential `/build` command that executes one plan's slices as Pi session-tree branches behind a **review-plus-verify** gate (an LLM `code-reviewer` AND a mechanical verify/test command), with a gated finalize that writes the plan's `SUMMARY`, advances `STATE`, and closes requirement traceability.

The redesign builds FROM SCRATCH on the Pi SDK and preserves the repo's identity: hard-gate custom tools that mechanically refuse, not skill-only conventions.

## Command surface

Each command takes an optional argument.

- `/init` (no arg) → scaffold empty `PROJECT.md` (mechanical, sole writer, refuse to clobber non-empty).
- `/init "<goal>"` → author/update `PROJECT.md` from the goal, reading REQ/ROADMAP/STATE for context. Lightweight, no review loop; re-runnable anytime. `/init` writes ONLY `PROJECT.md`.
- `/plan` (no arg) → scaffold empty `REQUIREMENTS`/`ROADMAP`/`STATE` (mechanical, sole writer, refuse to clobber non-empty).
- `/plan "<topic>"` → interview + ground (reading PROJECT/REQ/ROADMAP/STATE), produce ONE reviewed bundle (CONTEXT + PLAN + REQ/ROADMAP/STATE deltas). The planner itself decides whether `<topic>` is a new phase or extends an existing one and allocates the `NN-MM` id from ROADMAP/STATE — the user never types `NN-MM`. On a fresh repo (no ROADMAP yet) it bootstraps the initial REQ/ROADMAP/STATE from PROJECT + topic + interview, then plans the first phase's first plan, all in one gated bundle. On an existing repo it slots the topic into the roadmap, applying deltas.
- `/build` (no arg) → resolve and build the next plan from STATE/ROADMAP.
- `/build <planId>` → build a specific existing plan; warns-but-proceeds when earlier plans are unbuilt.

Note: the no-arg = mechanical-scaffold behavior applies to `/init` and `/plan` (their target docs may not exist yet). `/build` has no docs to scaffold; `/build` no-arg resolves the current plan from STATE.

## Constraints

- Build from scratch on the Pi SDK; no dependency on `pi-supergsd` or worktrees packages (reference-only for the branch mechanic).
- **Committed plugin runtime.** The framework depends on two already-adopted gotgenes Pi plugins (both already peer-dependencies; `@gotgenes/pi-subagents` already powers the `/plan` reviewer loop):
  - `@gotgenes/pi-subagents` (v18.x) provides the `subagent` / `get_subagent_result` / `steer_subagent` tools and the `.pi/agent/agents/<name>.md` custom-agent loading used for `plan-reviewer` and `code-reviewer`. **Hard constraint from this package: subagents cannot nest** — a recursion guard strips the subagent tools from any child session (`applyRecursionGuard` in `create-subagent-session.ts`), so a subagent can never spawn another subagent. This directly bounds the Phase-5 executor design (see the executor-primitive assumption): the `code-reviewer` subagent can only be spawned from a *top-level* turn (a session-tree branch turn), never from inside another subagent executor.
  - `@gotgenes/pi-permission-system` (v0.8.x) provides the enforcement layer for both reviewer read-only-ness and the `/build` protected-doc guard (see the two constraints below). It gates via a `tool_call` lifecycle hook using an `allow`/`deny`/`ask` state machine, per-agent frontmatter policy, JSONC path rules (`.pi/agent/pi-permissions.jsonc`, wildcard `*` matches recursively across `/`), and a trusted "security floor" that project-local rules cannot relax. It exposes no manual middleware object; enforcement is declarative config + its own hook, and cross-process subagent gating relies on the permission-forwarding env vars (`PI_AGENT_ROUTER_PARENT_SESSION_ID` etc.).
  Two further gotgenes packages are evaluated but NOT committed: `@gotgenes/pi-session-tools` is a spike-time convenience candidate for reading a branch leaf's persisted entries off disk (see the Phase-5 spike note); `@gotgenes/pi-subagents-worktrees` stays out of scope with the parallel-execution non-goal.
- Commands are bare actions limited to `/init`, `/plan`, `/build`.
- Execution runs **sequentially** as Pi session-tree branches (single active leaf); no parallel execution or worktrees.
- Hard-gate via custom tools that mechanically refuse (mirroring `validate-plan`/`finalize-plan`), not conventions.
- The build gate is **review-plus-verify**: a slice passes only when BOTH (a) a `code-reviewer` subagent returns blockers=0 and warnings=0-or-explicitly-accepted, AND (b) the project's mechanical **verify command** (typecheck/lint/test/format, resolved from the plan/project — see the verify-gate decision below) exits zero. The verify gate is a first-class hard gate, not a convention: a slice whose verify command fails is a blocker exactly like a review blocker, and `validate-change` will not persist a clean cycle for it. The *deterministic* portions of the change gate (out-of-scope path detection, reverse REQ coverage, verify exit code) are enforced mechanically in `validate-change`/`finalize-build`, independent of the reviewer's LLM judgment (see the fail-first decision below), so a rubber-stamping reviewer cannot pass an out-of-scope, uncovered, or verify-failing change.
- All protected-doc writes (`finalize-init`, `scaffold-docs`, `finalize-plan`, `finalize-build`) are **atomic**: write to a sibling temp file and `rename()` into place, so a crash mid-write can never leave a half-written `STATE.md`/`REQUIREMENTS.md`/`ROADMAP.md`/`PROJECT.md`/SUMMARY (the single source of truth for resume). Borrowed from gsd-core's `O_EXCL` STATE locking, minus the parallel-writer locking we don't need (sequential single-leaf execution).
- Authored prompts and agent bodies carry a **byte budget** enforced by test (`planner-starter.md`/`builder-slice.md`/`init-starter.md` and the two reviewer agents each under a per-file cap, e.g. 38KB). This is the proactive complement to the reactive heading-map bounded-section read: it keeps `/build`/`/plan` Turn-1 context lean as the framework grows. Borrowed from gsd-core's size-budget ratchets.
- New `code-reviewer` subagent mirrors `plan-reviewer` (read-only tools, rubric = the repo's `code-review-and-quality` skill, SAME review JSON shape) so `parseReviewResult` is reused unchanged; packaged in-repo under `.pi/agent/agents/` (project agents override global), so the framework ships self-contained. It may use a stronger/different default model than the builder when Pi supports per-agent model hints, because reviewer independence matters more than token price. Its rubric adopts obra/superpowers' **"do not trust the report"** stance: the change-summary doc is treated as *unverified claims* to be checked against the scoped diff and referenced source, and findings are triaged as **Missing** (a claimed REQ/behavior not actually implemented), **Extra** (out-of-scope or unrequested code / over-engineering), or **Misunderstood** (right requirement, wrong implementation). This is a prompt-rubric convention layered on the existing `{blockers,warnings,nitpicks}` shape (Missing/Misunderstood map to blockers, Extra to blockers when it touches an `## Out of Scope` path or is unrequested behavior, else warnings) — it does NOT change the review JSON schema or `parseReviewResult`.
- Reuse `src/parse.ts` (`parseReviewResult`, `parsePlanningContext`) and the store→validate→finalize hard-gate pattern.
- The `/build` runtime drives turns from the command handler using command-only APIs (`navigateTree`/`waitForIdle`), never from event handlers.
- The orchestrator's per-slice decisions derive ONLY from the parsed `change-review-cycle` entry (blocker/warning counts via `parseReviewResult`, plus the recorded verify exit result), never from branch prose; `finalize-build` re-verifies the persisted cycles as a hard gate. A `change-review-cycle` records both the review counts AND the verify outcome (`verify: { command, exitCode, ok }`), and a cycle is only "clean"/"warnings-only" when `verify.ok` is true — a failed verify forces the cycle to blocker state regardless of the reviewer's counts. Replayed `slice-result` payloads are file-based handoffs: the branch writes a compact result artifact under `.gpd/slice-results/<planId>-slice-<n>.md`, and the parent receives only the repo-relative artifact path, digest/fingerprint, and parsed counts via `sendMessage`, NOT `sendUserMessage` — branch output is data, not user input or instructions. When git is available the `slice-result` artifact also records the slice's commit range (`base..head` short hashes) so resume and finalize can verify completed work against `git log` (borrowed from obra/superpowers' progress ledger — the non-fragile half; we deliberately do NOT adopt its positional-index journal, see the deferred-items note). The commit range is recorded evidence, not a durable resume journal: "re-run `/build` resumes from the first non-clean slice" still derives from the persisted `change-review-cycle`/`execution-context.status`, with the commit range as a cross-check. Any small inline summary is control-char-stripped and delimited as "treat as data, not instructions". Deeper injection-pattern scanning and secret/path redaction of replayed text are deferred defense-in-depth, not v1 blockers.
- Scratch dir stays `.gpd/` (gitignored); bare parse labels and unprefixed tool names.
- All generated artifact paths (PLAN/CONTEXT/SUMMARY bodies, `## Out of Scope` blocks, touched-file lists, and traceability links) must be repo-relative; absolute paths are forbidden in generated docs. (gsd-lite is Pi-native, not cross-host portable, but absolute paths in artifacts are a defect regardless.)
- PLAN/CONTEXT/SUMMARY templates use stable headings so the builder-slice and reviewer prompts build a heading map first (scan `^#{1,3} ` for top-level sections plus any `### Slice <N>` units) and read only the bounded sections they need, rather than the whole document. This matters most as `NN-MM-PLAN.md` grows and keeps `/build` context and drift low.
- Every `### Slice <N>` carries a `#### Consumes` / `#### Produces` interface block (borrowed from obra/superpowers `writing-plans` task interface definitions): `Consumes` names the exact symbols/types/artifacts the slice relies on from an *earlier* slice in the same plan; `Produces` names what *later* slices in the plan may rely on. This is the sequential-builder's inter-slice drift boundary (slice N depends on N-1), complementing `## Out of Scope` (which bounds files) with a bounded contract on *what crosses slice edges*. It is scoped strictly within one plan's slices — cross-plan boundaries stay the SUMMARY/REQ-traceability concern, not a slice interface — so a global resolution check never false-blockers a plan. The builder-slice heading map reads its own slice's Consumes/Produces alongside `## Out of Scope`; `plan-reviewer` checks that each slice's `Consumes` resolves to some earlier slice's `Produces` (adequacy, semantic) and `finalize-plan` need not gate it mechanically (it is a within-doc reference the reviewer scores, not a protected-doc structural invariant like REQ coverage). Slices with no cross-slice dependency render an empty-or-"none" block, not a missing section.
- Tests use `node:test` via `tsx` with a mock `sessionManager` `{getBranch}`; `npm run verify` (typecheck + eslint + test + prettier) must stay green.

## Non-goals

- Parallel/concurrent execution, `pi-subagents-worktrees`, or git-merge-based parallelism.
- A bespoke CI system: the verify gate runs the *project's own* configured verify/test command (discovered during grounding, not a hardcoded language list); gsd-lite does not build a language-aware test runner, coverage tooling, or a merge queue of its own.
- A `/review` command (dropped; name not reserved).
- Auto-walking the entire roadmap in a single `/build` invocation (one plan per invocation).
- Cross-session resume of a `/build` interrupted mid-plan, or partial-slice resume.
- Cryptographic integrity for `reviewReadFingerprint` (cheap first/last-line anchor only, as today).
- Per-slice granular warning acceptance (a single plan-level `acceptWarnings` flag, mirroring planning).
- A public/user-facing workflow-scripting API or generic `gate`/`loopUntilDry` primitives exposed to users (any reviewed-loop helper stays strictly internal and private to the finalize/validate tools).
- Renaming the npm package.

## Assumptions

- The `/build` executor primitive is NOT assumed: both the reference repo (`pi-dynamic-workflows`, which spawns `SessionManager.inMemory()` isolated sessions) and gsd-lite's own working `/plan` (synchronous in-memory `subagent` loops) avoided the low-level session-tree branch APIs. Phase 5 therefore opens with a **comparative spike** that prototypes BOTH candidates — (a) the command-driven branch round-trip (`getLeafId` → `navigateTree` branch → `sendUserMessage` → `waitForIdle` → read branch leaf → `navigateTree` back → `sendMessage` replay) and (b) an in-memory subagent executor — and picks whichever proves the persistent-entry + review-loop needs with LESS machinery, before the per-slice review sub-loop is built on top. **Hard constraint on Candidate B (from `@gotgenes/pi-subagents`' recursion guard):** a subagent CANNOT spawn another subagent, so Candidate B's executor cannot invoke the `code-reviewer` as a *nested* subagent. Candidate B is therefore only viable if the review step runs as something OTHER than a nested subagent (e.g. the executor calls the reviewer model via a direct completion, or the orchestrator — not the executor — runs the `code-reviewer` subagent between executor turns). Candidate A does not hit this wall: the branch turn is a top-level turn and can spawn `code-reviewer` directly. This asymmetry is itself a spike finding to weigh — if driving the reviewer outside a nested subagent adds more machinery than the branch round-trip, Candidate A wins on both simplicity and correctness. The branch model's only distinctive payoff is a persistent leaf that accumulates `change-review-cycle` entries in the main session; if the in-memory executor's file-based `slice-result` handoff covers that need AND a non-nested reviewer path is clean, it competes on simplicity.
- Both review subagents are packaged in-repo under `.pi/agent/agents/` (`plan-reviewer.md` already moved there; `code-reviewer.md` authored there in Phase 4), so the framework ships self-contained without a global-install prerequisite. Pi resolves project agents at `.pi/agent/agents/<name>.md` and lets them override any global agent of the same name; both emit the same review JSON shape.
- `PROJECT.md` is a living doc written ONLY by `/init` (no-arg scaffolds empty; goal-arg authors/updates from the goal while reading REQ/ROADMAP/STATE for context). `/init` is re-runnable anytime and stays lightweight with no review loop.
- `/plan` owns the evolution of `REQUIREMENTS`/`ROADMAP`/`STATE` and the phase artifacts (`PROJECT.md` is `/init`-only). The planner decides new-phase-vs-existing and allocates the `NN-MM` id itself. Re-running `/plan` re-opens for revision and re-running `/build` re-executes — this re-triggerability is intended.
- The structural validation that a ROADMAP references only real REQ ids and STATE points at a real plan lives in the gated `finalize-plan` (which now owns those docs), not in `/init`.
- The executor turn needs `write`/`edit` (and `bash`) for source; protected docs (`PROJECT`/`REQUIREMENTS`/`ROADMAP`/`STATE` + `SUMMARY` paths) stay authoritative via the finalize/init tools plus a **`@gotgenes/pi-permission-system` path guard** active during `/build`: project-local `deny` rules on `write:.../docs/PROJECT.md`, `edit:.../docs/REQUIREMENTS.md`, etc. (and the `phases/**/NN-MM-SUMMARY.md` glob) block direct mutation and route those writes through the finalize/init tools. Using the permission system's declarative rules + trusted security floor replaces a hand-rolled `tool_call` guard; the finalize/init tools themselves are the sanctioned writers (they perform the atomic temp-file+rename, not a `write`/`edit` tool call, so the deny rules do not block them).
- The `code-reviewer` sub-loop must run inside an LLM turn (a subagent is an LLM-invoked tool). Because `@gotgenes/pi-subagents` forbids subagent nesting, the reviewer must be spawned from a TOP-LEVEL turn: the executor branch turn (Candidate A) owns implement+verify+review-until-clean for its slice and spawns `code-reviewer` directly; the orchestrator sequences slices and verifies via file-based `slice-result` entries on the parent branch. (If Candidate B is chosen, the reviewer cannot be a nested subagent — see the executor-primitive assumption.)
- `finalize-build` mirrors `finalize-plan`'s warning handling: a single plan-level `acceptWarnings` flag; blockers on ANY slice always refuse; `parseReviewResult` stays unchanged because acceptance lives in the finalize tool, not the parser.

## Slice review outcome semantics

A slice's latest `change-review-cycle` resolves to one of three states, and the orchestrator treats them differently:

- **clean** (blockers=0, warnings=0, `verify.ok`=true): slice succeeds; replay a clean file-based `slice-result` to the parent and advance to the next slice.
- **warnings-only** (blockers=0, warnings>0, `verify.ok`=true): NOT a stop condition. Replay a file-based `slice-result` carrying the outstanding warning counts/summary path to the parent and advance; the plan-level `acceptWarnings` decision is deferred to `finalize-build` (which refuses to write unless warnings are zero or `acceptWarnings` is set). Note `acceptWarnings` covers only reviewer *warnings* — a failed verify is never a warning and can never be accepted.
- **blockers** (blockers>0, `verify.ok`=false, or an unrecoverable failed cycle): a failed verify command counts as a blocker here. The executor branch revises and re-reviews, up to a hard cap of 3 implement→(verify+review)→revise cycles for the slice (on cycles 2 and 3 the branch turn works from the latest cycle's outstanding blockers, a compacted findings summary, not the full prior review history — see the `builder-slice.md` intra-slice cycle compaction note). If blockers remain after the 3rd cycle, the branch turn stops WITHOUT claiming the slice complete; the orchestrator sees a blockers-carrying latest cycle (or no clean cycle), sets `execution-context.status = blocked`, stops the loop, and surfaces the blocking slice to the user. Blockers are never carried past finalize. (The cap lives inside the builder-slice turn — the unattended loop — not on the orchestrator's one-pass slice sequencing. It is enforced by prompt convention in the first version, since the branch turn runs the framework's own prompt rather than adversarial input; a mechanical per-sliceId cycle-count cap in `validate-change` is a deferred hardening.)
- **interrupted** (the branch turn ended with `stopReason: 'aborted' | 'error'`, produced no parseable `change-review-cycle`, or a branch/reviewer `waitForIdle` exceeded its timeout without resolving): NOT a slice result. The orchestrator does NOT replay a `slice-result` to the parent and does NOT trigger finalize; it records `execution-context.status = paused` (aborted or timeout — the work is stalled, not corrupt, so re-running is safe) or `blocked` (error — no safe recovery) and surfaces the interrupted slice to the user. Re-running `/build` resumes from the first non-clean slice. The timeout wraps only the branch/reviewer `waitForIdle` waits; the finalize hard-gate tool is never timeout-wrapped, so its doc writes stay atomic.

## Resolved decisions

- **Verify gate (execution):** each slice must pass a mechanical **verify command** in addition to the `code-reviewer`. The command is resolved (not hardcoded) at build time from, in order: an explicit `verify:` field the plan may pin in `NN-MM-PLAN.md`, else a project-level default discovered during planner grounding and pinned into context (e.g. `npm run verify`), else — if none is discoverable — the plan MUST declare `verify: none` explicitly and the reviewer treats the absence as a warning (never a silent skip). The builder-slice turn runs the verify command via `bash` as the last step before `store-candidate-change`, captures `{command, exitCode}`, and passes it to `validate-change`, which records `verify: { command, exitCode, ok }` on the `change-review-cycle`. `validate-change` refuses to persist a clean or warnings-only cycle when `verify.ok` is false; a non-zero exit is a blocker the branch must fix within the 3-cycle cap. This is borrowed from gastown's hard, mechanically-enforced Refinery gates (`lint`/`test`/`build-check`), adapted to the sequential single-leaf model — gsd-lite gets failure isolation for free per slice, so it needs the gate but not gastown's batch-then-bisect. `finalize-build` re-verifies that every persisted slice cycle has `verify.ok=true` as a hard gate; it does NOT re-run the command (the executor branch owns execution), it re-checks the recorded exit result.
- **Review target (execution):** `store-candidate-change` writes a scratch change-summary doc `.gpd/candidate-changes/<id>.md` (touched files + what changed and why) and, when git is available, a scoped diff artifact (for example `.gpd/candidate-changes/<id>.diff`) that the reviewer reads alongside the referenced source files. `reviewReadFingerprint` anchors the summary doc. `validate-change` reuses `parseReviewResult` unchanged and persists a `change-review-cycle`.
- **"What to run next" pointer (borrowed from bigpowers `handoff.next_skill`/`survey-context`):** `STATE.md`'s current-pointer block carries an explicit human- and machine-readable *next action* line — the command + planId a cold-started session should run next (e.g. `next: /build 02-03`), plus the reason (planned-but-unbuilt, paused mid-build, blocked slice N). This is a legible answer to "what do I run next?" after any interruption, RENDERED from the existing status ledger + `execution-context.status` (planned→`/build`, paused/blocked→resume `/build` from the first non-clean slice, all-built→`/plan` the next roadmap item), NOT a new durable state buffer. `finalize-plan` and `finalize-build` update it as part of their existing STATE writes; every command reads it first. It complements — does not replace — the status-ledger-derived resume, giving one place that names the next step rather than requiring the user to infer it from the ledger.
- **`/build` scope:** one plan per invocation. `/build` with no arg resolves the current plan from `STATE.md`; `/build <planId>` targets a specific plan. After a plan's slices are all clean/warnings-only, it triggers finalize and stops (user re-invokes for the next plan).
- **Phase argument tradeoff:** `STATE.md` is a per-plan status ledger (pending → planned → built), not a single linear pointer. `/build <planId>` may jump ahead; the framework **warns but proceeds** when earlier plans are unbuilt, because downstream SUMMARY deliverables / REQ closures may be missing and STATE will show gaps. Flexibility is the goal.
- **Requirement-closure evidence (borrowed from gastown `completions`/`stamps` traceability):** when `finalize-build` closes REQ traceability in `REQUIREMENTS.md`, each satisfied REQ row records first-class *evidence* joining the requirement to its verification, not just a satisfied flag: `{ reqId, satisfiedBy: <planId>, summary: <SUMMARY path>, validatedBy: 'code-reviewer', verify: <command+ok>, evidence: <slice commit range base..head when git available> }`. This mirrors gastown's `wanted_id → completed_by → evidence → validated_by` chain (requirement → who did it → proof → who validated) but stays within the existing markdown traceability table + persisted-entry model — no SQL ledger, no new durable state buffer. The fields are RENDERED by `finalize-build` from data it already has (the persisted `change-review-cycle` entries carry the reviewer outcome + verify result; the `slice-result` artifacts carry the commit range), so this is a legible view over existing sources of truth, not a second one. It makes the reviewer/verify link auditable in `REQUIREMENTS.md` itself rather than only inferable from scattered cycles.
- **Finalize wiring (execution):** the `/build` command handler runs the slice loop; once every slice's replayed cycle is clean or warnings-only (no blockers), the handler triggers `finalize-build` via a terminal prompt so the hard-gate tool re-verifies the persisted cycles before writing. `finalize-build` refuses unless every slice's latest cycle has zero blockers, `verify.ok=true`, and (zero warnings or `acceptWarnings`), then writes `NN-MM-SUMMARY.md`, updates `STATE.md`, updates `REQUIREMENTS.md` traceability, and appends a `build-finalized` entry. The executor has `write`/`edit` for source but NOT for the protected docs.
- **Accepted-warning model:** `finalize-build` takes a single `acceptWarnings` boolean exactly like `finalize-plan`; blockers on any slice can never be accepted. No parser change.
- **`/init` gate:** lightweight, no review loop. `finalize-init` is the sole writer of `PROJECT.md`: no-arg writes an empty template (refuse to clobber non-empty); goal-arg authors/updates content. It reads REQ/ROADMAP/STATE for context during authoring but writes only `PROJECT.md`.
- **`/plan` doc writes:** substantive and gated. `/plan` may add/refine REQ ids and restructure ROADMAP; those edits are part of what `plan-reviewer` scores and are written only by the reworked `finalize-plan` alongside CONTEXT + PLAN, plus a STATE status update to `planned`. `finalize-plan` also owns the structural validation (ROADMAP refs resolve to REQ ids; STATE pointer references a real plan).
- **Bidirectional requirement coverage (scoped):** coverage is enforced in two complementary places, both scoped strictly to the REQ ids a plan *claims* (never the whole `REQUIREMENTS.md` backlog — the ROADMAP spreads REQs across many plans, so a global check would false-blocker every plan). (1) Semantic: the `plan-reviewer` rubric blocks when a claimed REQ has no slice, a slice has no claimed REQ, a slice references but does not actually satisfy its REQ, a slice's `#### Consumes` does not resolve to an earlier slice's `#### Produces`, or the finalized CONTEXT/PLAN contains an unresolved placeholder (`TODO`/`TBD`/`FIXME`/`???`) or an internal contradiction (anti-placeholder check borrowed from obra/superpowers spec/plan review). (2) Mechanical: `finalize-plan` fails closed on the *reverse* direction as a hard gate — every REQ id the plan claims must map to ≥1 slice — complementing the existing forward gate (every slice's REQ ids must resolve). Reviewer catches adequacy (including the placeholder and Consumes/Produces checks); finalize catches structural gaps.
- **Planner grounding refinements:** `planner-starter.md` adds (a) an anti-greenfield grounding step that reads the project's *stack-appropriate* authoritative setup/config/build files (no hardcoded language list) and treats them as ground truth into `repoFindings`, and (b) a transient known-unknowns triage during grilling that drains each unknown — including external dependencies — into ask-user / `doc-lookup` / `codebase-explorer` / `deferredItems`, landing resolved ones in `repoFindings`/`assumptions`/`constraints`. Known-unknowns are deliberately NOT a pinned planning-context field (resolving them would trip the context-drift guard); external dependencies are folded into existing fields rather than adding a new schema field. It also adds (c) a **surface-alternatives** convention (borrowed from obra/superpowers `brainstorming`): when a genuine design fork exists, the planner presents 2–3 approaches with tradeoffs and a specific recommendation before pinning the plan, rather than latching onto the first solution. This is a grilling-phase prompt convention, not a pinned field or a gate.
- **Package legitimacy triage (borrowed from gsd-core slopcheck):** the known-unknowns triage step above explicitly covers *newly introduced third-party dependencies*: an unrecognized package, an open version range where a pin is expected, or a name that looks like a typosquat is drained to ask-user or `doc-lookup` and, if unresolved, recorded as a `deferredItem` (never silently assumed legitimate). Resolved packages land pinned in `assumptions`/`constraints`. This is a prompt-convention grounding rule aligned with the repo's own dependency safety guardrails, NOT a new schema field or a mechanical supply-chain scanner (gsd-core's full slopcheck/`[ASSUMED]` tagging is out of scope for v1).
- **Revise-in-place (planning):** on a blocked or warning planning review, the planner revises the prior stored candidate bundle in place using the reviewer critique, then re-stores it for a fresh id — it does NOT regenerate unrelated sections of the bundle from scratch. This is a prompt convention that complements (does not replace) the context-drift pin and `reviewReadFingerprint`: those guard the pinned planning-context JSON and prove the reviewer read the stored bytes, but neither stops a from-scratch regeneration of the CONTEXT+PLAN+deltas bundle while the context JSON stays byte-identical. The build side already has the equivalent via the intra-slice cycle compaction note.
- **`NN-MM` allocation:** the planner allocates the id from ROADMAP/STATE (next sequential plan within the target phase, or a new phase number); the user never types it. `/build <planId>` references an existing allocated id.
- **No-arg scaffolding:** `/init` no-arg → `finalize-init` empty mode (PROJECT.md). `/plan` no-arg → a mechanical `scaffold-docs` tool writes empty `REQUIREMENTS`/`ROADMAP`/`STATE` templates (refuse to clobber non-empty). Both are mechanical refuse-to-clobber writers, no review.
- **Control flow:** the `/build` command handler runs the whole sequential loop in its own call frame using command-only `navigateTree`/`waitForIdle`.
- **Entry/tool naming (aligned to `/build`):** entries `stored-candidate-change`, `change-review-cycle`, `execution-context`, `build-finalized`; tools `finalize-init`, `scaffold-docs`, `store-candidate-change`, `validate-change`, `finalize-build`. Planning entries/tools unchanged.
- **Reviewer agents are declaratively read-only:** both `plan-reviewer.md` and `code-reviewer.md` declare read-only tools + `inherit_context: false` in their frontmatter, and each carries a `@gotgenes/pi-permission-system` per-agent policy block that denies `write`, `edit`, and `special: external_directory` (the permission system resolves the active agent via the `<active_agent name="..."/>` prompt tag that `@gotgenes/pi-subagents` prepends, and its trusted floor prevents a project-local rule from relaxing the deny). Each reviewer also ships an example output blob that must round-trip through `parseReviewResult`. Structural tests (Phase 4) enforce both, so a reviewer can never silently gain mutation rights or drift from the shared review-JSON schema.
- **Fail-first mechanical change gate (borrowed from gsd-core prohibition-enforcement):** the deterministic parts of the change gate do not rely on the LLM reviewer's judgment and are proven with a fail-first fixture pair. Specifically, `validate-change`/`finalize-build` mechanically detect (a) a candidate change whose touched-file list includes any path matching the plan's `## Out of Scope` block, (b) a claimed REQ id with no covering slice, and (c) a recorded `verify.ok=false` (non-zero verify exit) — each is a hard blocker. Following gsd-core's pattern, the tests assert BOTH directions: a *violation fixture* (an out-of-scope touch / an uncovered REQ / a failing-verify cycle) MUST be blocked, and a *clean control fixture* MUST pass, so the gate can never silently always-pass. The LLM `code-reviewer`'s adequacy judgment stays a prompt-convention + manual-smoke concern as before; only the mechanically decidable checks get the fixture-pair proof.
- **Agent descriptions are trigger-only:** command/agent/skill descriptions say only WHEN to load or invoke the artifact, not a compressed workflow summary. Workflow steps live in the full prompt/agent body so the model cannot shortcut from metadata and skip required gates.

## Artifact model

```
docs/
  PROJECT.md              # identity / mission (living; written only by /init)
  REQUIREMENTS.md         # REQ-NN list + traceability table (living; /plan + finalize-build)
  ROADMAP.md              # phases -> REQ ids, ordered plans per phase (living; /plan)
  STATE.md                # per-plan status ledger + current pointer (living; /plan + finalize-build)
  phases/
    NN-name/
      NN-CONTEXT.md        # phase context (written by finalize-plan)
      NN-MM-PLAN.md        # plan with enumerable slices + REQ ids (written by finalize-plan)
      NN-MM-SUMMARY.md     # execution deliverable (written by finalize-build)
```

Doc ownership (sole gated/mechanical writers):

- `PROJECT.md` → `finalize-init` only.
- `REQUIREMENTS`/`ROADMAP`/`STATE` (empty scaffold) → `scaffold-docs`.
- `CONTEXT`/`PLAN` + REQ/ROADMAP deltas + STATE `planned` → `finalize-plan`.
- `SUMMARY` + STATE `built`/advance + REQ traceability closure → `finalize-build`.

Each living doc carries a small machine-parseable fenced block that the commands read/update; the rest is human-readable prose. Templates live under `prompts/` (or a `templates/` dir) and are used by `/init`, `/plan`, and the finalize/scaffold tools.

### Phase 5 — Sequential `/build` runtime + per-slice review sub-loop

- **Slice 1 (COMPARATIVE SPIKE):** prototype BOTH executor candidates and pick one before building the review sub-loop on top. **Candidate A (branch round-trip):** in the `/build` command handler, `getLeafId` → `navigateTree(parent,{summarize:false})` → `sendUserMessage(prompt)` → `waitForIdle()` → read branch leaf's last assistant text → `navigateTree(parent)` → `sendMessage(result)`; the branch turn is top-level so it spawns `code-reviewer` directly. **Candidate B (in-memory subagent executor):** an isolated executor session (mirroring `/plan`'s existing `subagent` pattern and pdw's `SessionManager.inMemory()`) that owns implement+verify+review-until-clean, handing results back as a file-based `slice-result` payload. **Candidate B constraint:** the executor is itself a subagent, and `@gotgenes/pi-subagents`' recursion guard forbids it from spawning the `code-reviewer` as a nested subagent — so Candidate B must run the reviewer via a non-nested path (orchestrator-driven reviewer subagent between executor turns, or a direct reviewer-model completion), which is part of what the spike measures. Compare on: does the persistent-leaf entry accumulation buy anything the file-based handoff doesn't; the cost of the non-nested reviewer path for B vs. A's direct spawn; total machinery/LOC; and reliability of `waitForIdle` timeouts. **Also evaluate in the spike:** whether `@gotgenes/pi-session-tools`' `readSessionFileEntries(file, {types})` lets the orchestrator read a branch leaf's persisted `change-review-cycle` entries off disk instead of `navigateTree`-ing back to read them — noting its caveats (no package `exports` map → deep-import fragility; JSONL read-while-write race risk). It is an optional convenience layered on the file-based `slice-result` contract (which stays primary), NOT a committed dependency; adopt only if the spike shows it removes navigation round-trips cleanly. Record the decision + rationale, then build the rest of Phase 5 on the winner. Gate the rest of the phase on ONE candidate proving out; if neither does, stop and surface. (The subsequent Phase-5 slices below are written against the branch vocabulary; if Candidate B wins, the same slice-loop/replay/timeout semantics apply with the branch swapped for the in-memory session — the file-based `slice-result` contract is deliberately primitive-agnostic.)
- `/build [planId]` resolves the target plan from `STATE.md` (or the arg), warns-but-proceeds on out-of-order builds, and persists an `execution-context` entry `{planId, phaseId, slices, reqIds, parentLeafId, status}` (status starts `active`).
- Throughout the loop the handler maintains a TUI status line via `ctx.ui.setStatus('gpd-build', `gpd · ${planId} · slice ${i + 1}/${slices.length}`)`, updated per slice and cleared with `ctx.ui.setStatus('gpd-build', undefined)` in a `finally` so it clears on clean exit, abort, and error alike. (`setStatus(key, text)` is confirmed on the full `ExtensionCommandContext.ui`; the command frame that owns the loop is its natural owner during the long branch/review passes.)
- Alongside the single status line the handler renders a **bounded rolling progress log** (last N events, e.g. 40) covering slice start / implementation-complete / review-outcome / revision-cycle / clean transitions. This is a DISPLAY-ONLY render derived from the already-persisted `change-review-cycle` entries (and the current slice index) — NOT a new durable state buffer. The single line collapses the up-to-3 implement→review→revise cycles per slice; the rolling log surfaces that history without competing with the entry model as a second source of truth. Cleared with the status line in the same `finally`.
- Each branch/reviewer `waitForIdle` is wrapped with a **timeout** (config/const, generous default) so a hung subagent or stalled model cannot hang the command frame indefinitely. A timeout is treated as **interrupted → `execution-context.status = paused`** (matching the `aborted` path per the Slice review outcome semantics): no `slice-result` replay, no finalize trigger, surface the stalled slice, and re-running `/build` resumes. The timeout deliberately does NOT wrap the finalize terminal prompt / `finalize-build` doc writes, which must stay atomic.
- Orchestrator loop (command handler, primitives injected for testability): for each slice, branch → send the builder-slice prompt → `waitForIdle` → read the branch's latest `change-review-cycle` and resolve it per the Slice review outcome semantics → navigate back → `sendMessage` a file-based `slice-result` payload onto the parent (repo-relative result artifact path + digest + parsed counts; no raw branch transcript). Clean and warnings-only advance the loop; unresolvable blockers stop it and surface to the user. If `waitForIdle` resolves with the branch leaf's last assistant `stopReason` of `aborted` or `error`, or with no parseable `change-review-cycle`, the orchestrator treats the slice as **interrupted**: it does NOT replay a `slice-result` and does NOT trigger finalize, writes `execution-context.status = paused` (aborted) or `blocked` (error), and stops. (No retry-classification machinery: turns are driven explicitly from the command frame, so there is no auto-continuation to deduplicate — stop-and-surface is the whole policy.)
- `store-candidate-change` + `validate-change` tools (mirror `store-candidate-plan`/`validate-plan`; reuse `parseReviewResult` + fingerprint against the change-summary doc). `validate-change` additionally takes the recorded `{command, exitCode}` from the slice's verify run, computes `verify.ok`, and refuses to persist a clean/warnings-only `change-review-cycle` when `verify.ok` is false (a failing verify is a mechanical blocker, independent of the reviewer's counts). Add a small slice-result writer/serializer under `.gpd/slice-results/` so parent replay stays path+digest based rather than prose-heavy; the writer records the slice's git commit range (`base..head` short hashes) when git is available, as verifiable resume evidence (not a durable journal).
- `prompts/builder-slice.md`: instructs the executor branch turn to first build a heading map of `NN-MM-PLAN.md` / `NN-CONTEXT.md` (scan `^#{1,3} ` plus `### Slice <N>` units) and read only the bounded sections it needs (its slice, its slice's `#### Consumes` / `#### Produces` interface block, `## Out of Scope`, covered REQ ids, the pinned `verify:` command) rather than the whole doc, then implement the slice, then run the resolved **verify command** via `bash` and capture `{command, exitCode}`, then `store-candidate-change` → `code-reviewer` → `validate-change` (passing the verify result), revising until BOTH verify passes AND blockers are cleared (warnings may remain for the plan-level acceptWarnings decision), up to the hard cap of 3 implement→(verify+review)→revise cycles from the Slice review outcome semantics — after the 3rd failed cycle the turn stops without claiming the slice complete rather than looping unattended. The prompt requires the branch turn to (a) treat any diff touching a path in the plan's `## Out of Scope` block as a blocker to resolve, (b) audit each REQ id the slice covers individually — confirming the change actually satisfies it — before claiming the slice complete, rather than asserting completion in aggregate, and (c) use a verification-before-completion rule: before saying a slice is done, run the verify command fresh and read its actual exit result, and avoid "should"/"seems"/"probably" claims. When the plan pins `verify: none`, the branch turn skips the command and records `verify.ok=true` with a null command, and the reviewer is told the absence is a warning-worthy gap.
  - **Intra-slice cycle compaction:** on revise cycles 2 and 3 the prompt directs the branch turn to work from the *latest* persisted `change-review-cycle`'s outstanding blockers (a compacted findings summary), not the full prior review history. Subagents (`codebase-explorer`/`doc-lookup`/`code-reviewer`) already run `inherit_context: false`, so only their returned results — not their internal reasoning — enter the builder turn; this convention keeps stale full review blobs from earlier cycles out of the working context. It is a prompt convention, not a new durable state buffer: the persisted cycles remain the source of truth, and cross-slice/parent compaction is already handled structurally by the branch-per-slice model (only a compact `slice-result` is replayed to the parent).
- Protected-doc guard active during `/build` via `@gotgenes/pi-permission-system`: project-local `deny` rules (in `.pi/agent/pi-permissions.jsonc`) block direct `write`/`edit` to protected docs (PROJECT/REQUIREMENTS/ROADMAP/STATE + the `phases/**/NN-MM-SUMMARY.md` glob), routing them through the finalize/init tools; the permission system's trusted security floor keeps a project-local rule from relaxing the deny. Add a test asserting the rules block a direct `write`/`edit` to each protected path while permitting source paths.
- **Verify:** tests for the orchestrator loop with fake primitives (clean / warnings-only advance, blockers stop; a slice whose latest cycle still carries blockers after the cap sets status blocked and surfaces the slice; a slice whose latest cycle has `verify.ok=false` is treated as a blocker and never advances; interrupted `aborted`/`error`/no-cycle sets status paused/blocked and does NOT replay a slice-result or trigger finalize; a branch/reviewer `waitForIdle` that exceeds its timeout sets status paused, does NOT replay a slice-result or trigger finalize, and surfaces the stalled slice; branch/return/replay sequencing; replay payloads contain only a repo-relative slice-result path, digest, parsed counts, and (when git is available) the slice's commit range; the status line is set per slice and cleared on clean/abort/error exit; the bounded rolling progress log renders from persisted cycles and is capped to N events), `store-candidate-change`/`validate-change` mirroring the plan-tools tests (including the fail-first fixture pair for out-of-scope paths AND for a failing-verify cycle: an out-of-scope-path touch and a `verify.ok=false` cycle MUST each block, and a clean control MUST pass), slice-result writer tests, and the permission-system rules blocking direct `write`/`edit` to protected paths while permitting source paths; manual smoke for the real spike round-trip; `npm run verify` green.

### Phase 6 — `finalize-build`: SUMMARY + STATE + REQ closure

- After the Phase-5 loop confirms all slices are clean or warnings-only (no blockers), the `/build` command handler triggers `finalize-build` via a terminal prompt (the hard-gate tool re-verifies the persisted cycles before writing). The terminal prompt requires a per-REQ audit: for each covered REQ id, confirm the finalized SUMMARY and the persisted slice cycles actually satisfy it before writing, rather than asserting plan completion in aggregate.
- `finalize-build` refuses unless every slice's replayed `slice-result`/`change-review-cycle` on the parent branch has zero blockers, `verify.ok=true`, and (zero warnings or `acceptWarnings`); it takes a single plan-level `acceptWarnings` flag (mirroring `finalize-plan`; blockers on any slice always refuse, and a failing verify is never acceptable). It then writes `NN-MM-SUMMARY.md` (RENDERING the `## Attempts / Blockers` section — including the per-cycle `Verify` column — from the persisted `change-review-cycle` entries, a derived view, not a hand-maintained ledger), updates `STATE.md` (mark plan `built`, advance the pointer to the next unbuilt plan and re-render the current-pointer `next:` action line — all-built → `/plan` the next roadmap item, else `/build` the next unbuilt plan — from the updated ledger + `execution-context.status`, surface gaps for out-of-order builds), updates `REQUIREMENTS.md` traceability (mark covered REQ ids satisfied and RENDER the closure-evidence fields — `satisfiedBy`/`summary`/`validatedBy: code-reviewer`/`verify`/`evidence` commit range — per the requirement-closure-evidence decision, all derived from the persisted cycles + slice-result artifacts, not re-authored), and appends a `build-finalized` entry.
- Clears `.gpd/candidate-changes/` and `.gpd/slice-results/` on success (mirroring the planning cleanup); retains them on failure.
- `finalize-build` fails closed if a covered REQ id does not resolve.
- **Verify:** tests that finalize writes SUMMARY, renders the `## Attempts / Blockers` section (with its `Verify` column) from the persisted `change-review-cycle` entries, advances STATE, advances STATE and re-renders the current-pointer `next:` action line from the updated ledger, closes REQ traceability including the rendered closure-evidence fields (`satisfiedBy`/`summary`/`validatedBy`/`verify`/`evidence`), refuses when any slice has blockers, refuses when any slice has `verify.ok=false` (never acceptable via the flag), accepts warnings-only slices only with the flag while always refusing blockers, refuses on unresolved REQ ids, and (fail-first fixture pair) blocks a change touching an `## Out of Scope` path while passing a clean control; a crash-safety test that a protected-doc write is atomic (temp-file + rename, no half-written doc on a simulated mid-write failure); `npm run verify` green.

- **Parallel slice execution with worktree isolation (deferred, documented future direction):** `pi-dynamic-workflows` runs independent subagents in parallel behind a concurrency limiter (default cap 8) and, for parallel file edits, isolates each in a temporary git worktree (`git worktree add -b pi/wf/<id>`, force-deleted on completion) so concurrent agents cannot clobber each other's workspace. gsd-lite v1 is deliberately strictly sequential (single active leaf, review-gate depends on slice ordering, worktrees listed as an explicit non-goal), which is far simpler and correct for a local single-user review-gated build. But for a build phase with genuinely independent slices, parallel-slices-in-worktrees is the single biggest capability pdw has that v1 gives up. Revisit ONLY if sequential build latency becomes a real pain and slice independence can be declared reliably; it would add worktree lifecycle management, a merge/reconcile step, and a review-ordering story that the sequential single-leaf model avoids entirely.

- **Retry/stop policy when a slice cannot clear blockers: the builder-slice turn revises up to a hard cap of 3 implement→review→revise cycles, then stops and surfaces the blocking slice (`execution-context.status = blocked`). Enforced by prompt convention in v1; a mechanical per-sliceId cycle-count cap in `validate-change` is deferred hardening.
- Cross-session / mid-plan resume of an interrupted `/build` (default: re-running `/build` reads `execution-context.status`; a `paused`/`blocked` context restarts from the first non-clean slice).
- Whether the `@gotgenes/pi-permission-system` path-rule guard vs. a documented convention is the final enforcement for protected docs (plan adopts the permission-system rules; open to simplification if they prove heavy). Also open: whether subagent-side permission forwarding (which needs `PI_AGENT_ROUTER_PARENT_SESSION_ID` and a shared runtime dir) is required for `/build`, or whether the reviewer's declarative deny + the source-only executor policy suffice without interactive forwarding.
- Exact machine-parseable format of the STATE status block and PLAN slice list (defined in Phase 1; may be refined during Phases 3/5).
- Whether `scaffold-docs` and `finalize-init` should later merge into one mechanical scaffold tool (kept separate for now; PROJECT vs the REQ/ROADMAP/STATE trio).
- Robustness of review-JSON extraction: `extractJsonBlock` already strips ```` ```json ```` fences and `parseReviewResult` applies strict schema validation on top, and the reviewer runs as a subagent inside a turn we prompt on both ends (so a fenced block can be required). Borrowed from `pi-dynamic-workflows`' schema recovery (its `maxSchemaRetries` re-prompts a schema-failing agent with the toolset restricted to `structured_output`, then a regex/JSON fallback, before throwing): a **v1 one-retry recovery** is pulled forward — when `parseReviewResult` fails on a reviewer response, re-prompt the reviewer once demanding ONLY the fenced review-JSON block, and only then treat the cycle as an unparseable failure. The narrow outermost-balanced-brace fallback for unfenced JSON stays a deferred opportunistic one-line change in `extractJsonBlock`.
- **Reusable reviewed-loop helper (deferred to Phase 7, conditional):** `pi-dynamic-workflows` ships a reusable `gate(thunk, validator, {attempts})` primitive — run a generator, validate, feed the validator's feedback into the next attempt, cap at N. gsd-lite's planning revise-loop (`store-candidate-plan` → `plan-reviewer` → `validate-plan`, revise-in-place until clean) and the Phase-5 intra-slice loop (`store-candidate-change` → `code-reviewer` → `validate-change`, revise until blockers clear, hard cap 3) are structurally the same generate→review→revise-until-clean shape. **Do NOT pre-design a shared abstraction.** Write both loops concretely (planning in Phase 3, slice in Phase 5). After Phase 5, if the two loops share genuine structure — the store/review/validate sequencing, the cap, the compacted-feedback-into-next-cycle step — extract an internal helper (e.g. `runReviewedCycle({ store, review, validate, cap, compactFeedback })`) used by both. This is an *internal* refactor to reduce bugs across the hard-gate tools; it is explicitly NOT public workflow scripting, and it must not change the persisted entry model (the `change-review-cycle` / `plan-review-cycle` entries stay the single source of truth). Skip the extraction if the concrete loops turn out to diverge more than they share.
- **Plan/doc integrity attestation (deferred hardening, not v1):** `planning-with-files` locks a plan hash and detects external tampering. This guards a DIFFERENT threat than the existing `reviewReadFingerprint` + context-drift pin — those prove *the reviewer read the stored bytes* within a run, whereas attestation would detect *out-of-band tampering of the docs on disk between runs*. Candidate targets if built: `NN-MM-PLAN.md`, `REQUIREMENTS.md`, `STATE.md`. For a local single-user framework this is low priority; v1 stays on the cheap first/last-line anchor by explicit decision (see the `reviewReadFingerprint` non-goal). Revisit only if multi-user or untrusted-doc-source workflows become real.
- **Deterministic slice resume cache (deferred, likely won't build):** `pi-dynamic-workflows` journals agent results (keyed by call index + a params hash, replaying the longest unchanged prefix) because its intermediate work is ephemeral — it lives only in script memory and an in-memory shared store, so the serialized journal is the sole durable record across runs. **Confirmed fragility (reason not to borrow):** pdw keys the cache on a positional `callSeq` index, so ANY upstream change — even adding a log line or an extra loop iteration — moves `state.firstMiss` and invalidates the entire downstream cache even for calls whose inputs are unchanged. gsd-lite's artifact-first model already persists completed slice work (source edits on disk + `change-review-cycle` entries + `execution-context.status`), so "re-run `/build` resumes from the first non-clean slice" already reuses prior work without a journal AND without the positional-replay brittleness. A hash-keyed cache (`sliceSpecHash`, dependency summary hash) would only add value for detecting a *clean slice whose PLAN spec changed mid-build* — an edge case that conflicts with the "partial-slice resume" non-goal and the repeated "no new durable state buffer" principle. Revisit only if mid-build plan edits become a real workflow; v1 stays sequential, no worktrees, no journal.
- **Separate cached research artifact (deferred, borrowed observation from gsd-core):** gsd-core splits research (`gsd-phase-researcher` → `XX-RESEARCH.md`) from planning (`gsd-planner`) so re-planning a phase does not re-run research. gsd-lite instead folds grounding into `planner-starter.md`'s anti-greenfield read + known-unknowns triage, landing results in the pinned `repoFindings`/`assumptions`/`constraints` and rendering them into `NN-CONTEXT.md`'s `## Findings`/`## Decisions`/`## Deferred Unknowns`. This is leaner and correct for the current scale. Revisit a distinct cached `NN-RESEARCH.md` (read as ground truth by the planner, not re-derived on re-plan) ONLY if planning-context grounding becomes expensive enough that re-planning routinely re-does heavy research; v1 keeps grounding inline.
